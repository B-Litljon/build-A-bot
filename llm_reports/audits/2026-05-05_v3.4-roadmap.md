# Project Audit & Roadmap: Build-A-Bot V3.4/V4.0
**Date:** 2026-05-05
**Status:** CRITICAL - SUB-OPTIMAL
**Auditors:** Senior Dev / Picky Investment Broker

---

## 1. Executive Summary: The Broker's Verdict
From an investment perspective, this bot is a **high-risk liability** in its current state. While the underlying ML architecture (Angel/Devil Meta-Labeling) is conceptually sound, the operational implementation is fragile. We are essentially flying a sophisticated jet with a single, massive, over-complicated circuit board (`LiveOrchestrator`) and no flight recorder. 

**Key Risks:**
- **Inconsistent Risk Guardrails:** Risk logic is duplicated and partially hardcoded in the orchestrator, bypassing the dedicated `RiskManager`.
- **Operational Blindness:** The "Kill Switch" depends on a missing `drift_report.json`. We are trading without a safety net.
- **Zero Test Coverage:** No unit tests for the execution engine. One logic error in the state machine results in 100% loss of capital.

---

## 2. Technical Audit: The Senior Dev's Deep Dive

### A. Architectural Debt: The "God Object"
`LiveOrchestrator.py` is a 2400-line monstrosity. It violates the Single Responsibility Principle (SRP) at every turn. It manages:
- WebSocket Lifecycle (Stock, Crypto, Trade)
- Rich CLI Rendering
- Feature Engineering (Warm/Cold paths)
- Risk Management & Position Sizing
- Order Submission & State Persistence
- Symbol State Machine

**Consequence:** A bug in the "Dashboard" code can (and will) crash the trading loop. Maintenance is a nightmare.

### B. Shadow Risk Logic
The project has a `RiskManager.py`, yet `LiveOrchestrator._submit_entry_order` reimplements its own sizing and SL/TP logic. 
- **Fault:** `MIN_SL_PCT` and `ACCOUNT_RISK_PER_TRADE` are defined in both places. If they diverge, the "Shield" is broken.
- **Fault:** The `RiskManager` class is instantiated but barely used in the live path.

### C. Data & ML Integrity
- **Drift Detection:** `drift_report.json` is missing. The orchestrator defaults to a hardcoded `ATR_KILL_SWITCH_THRESHOLD` which may be invalid for current market regimes.
- **Feature Skew:** The "Warm Path" literal injection in `LiveOrchestrator` is a performance optimization that introduces a high risk of inference/training skew.
- **V4.0 Half-Measure:** The Day Trade model (5m) is documented but the "Integration Plan" in `action_plan.md` is mostly unchecked. It's a "ghost" feature.

### D. Operation & Tooling
- **Root Clutter:** 8+ `backtest_*.py` files in the root. No unified backtesting framework.
- **Dependency Bloat:** `Pipfile` contains both `scikit-learn` and `lightgbm`, but only RF (sklearn) seems to be the focus.

## 3. The Roadmap (2026 Q2-Q3)

### Phase 1: Deconstruction (Next 2 Weeks)
- **[ ] Decouple Orchestrator:** Break `LiveOrchestrator` into:
    - `StreamManager`: Handles Alpaca WS connections.
    - `ExecutionEngine`: Pure state machine for trades.
    - `UIBridge`: Handles Rich dashboarding (isolated).
- **[ ] Centralize Risk:** Remove all sizing/bracket logic from `LiveOrchestrator`. `ExecutionEngine` must call `RiskManager` for every trade.
- **[ ] Unified Logging:** Standardize all logs to JSON format for better downstream analysis.

### Phase 2: Fortification (Next 4 Weeks)
- **[ ] Core Test Suite:** 80%+ coverage for `RiskManager`, `FeaturePipeline`, and `SymbolState` transitions.
- **[ ] Drift Recovery:** Fix the `DriftEvaluator` to reliably produce `drift_report.json`. The bot should refuse to start if this file is older than 24h.
- **[ ] Backtest Registry:** Move all backtests to `tests/backtests/` and create a single `run_backtest.py` with parameter overrides.

### Phase 3: Evolution (Next 8 Weeks)
- **[ ] V4.0 Full Integration:** Complete the 5m Day Trading pipeline. 
- **[ ] Multi-Model Hot-Swapping:** Allow the bot to switch between Scalper (1m) and Day Trader (5m) modes based on regime detection.
- **[ ] Shadow Mode:** Implement a "Paper-on-Live" mode where new models run in parallel with the production model without executing orders.
